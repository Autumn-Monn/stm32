# 智能盆栽养护系统渐进式开发任务清单

## 1. 规划目标

这份清单用于把当前项目拆成一系列**可以逐步完成、逐步验证、逐步积累代码经验**的小任务。

整体原则只有三条：

1. 先做最容易验证的功能，再做复杂外设与联动逻辑。
2. 先在 `main.c` 的 `USER CODE BEGIN/END` 区域完成最小可运行版本，再逐步抽离成独立模块。
3. 每完成一个阶段，都保留一份“本阶段 AI 指令 + 编写逻辑说明 + 验收结果”，方便后续继续接力开发。

---

## 2. 当前开发边界

本阶段是**项目规划阶段**，不是一次性把所有功能写完。

已知你当前掌握的理论基础主要包括：

- GPIO
- IIC
- UART
- 中断优先级

因此开发路线要从**点灯、按键、蜂鸣器、继电器**这类最直接的 GPIO 任务开始，再进入：

- 外部中断
- 串口调试
- ADC 采样
- I2C 外设
- 单总线温度采集
- 系统联动与状态机

---

## 3. 与 CubeMX 配合的代码存放原则

### 3.1 绝对优先使用的安全区域

CubeMX 重新生成代码后，通常会保留以下区域中的内容：

- `/* USER CODE BEGIN Includes */`
- `/* USER CODE BEGIN PTD */`
- `/* USER CODE BEGIN PD */`
- `/* USER CODE BEGIN PM */`
- `/* USER CODE BEGIN PV */`
- `/* USER CODE BEGIN PFP */`
- `/* USER CODE BEGIN 0 */`
- `/* USER CODE BEGIN 1 */`
- `/* USER CODE BEGIN 2 */`
- `/* USER CODE BEGIN 3 */`
- `/* USER CODE BEGIN 4 */`

### 3.2 推荐使用方式

- 阶段前期验证代码：
  先写在 `Core/Src/main.c` 的 `USER CODE` 区域里，方便快速跑通。
- 阶段完成后稳定下来的功能：
  再逐步抽离到独立文件，例如：
  - `Core/Inc/app_led.h`
  - `Core/Src/app_led.c`
  - `Core/Inc/app_key.h`
  - `Core/Src/app_key.c`
  - `Core/Inc/app_sensor.h`
  - `Core/Src/app_sensor.c`

### 3.3 中断相关原则

- 中断服务函数要短。
- 中断里只做“置标志位、记录事件、更新时间戳”这类快速操作。
- 真正的业务处理放到 `while(1)` 主循环中。

## 阶段 0：建立最小开发规范

### 目标

先不做业务功能，先把项目的“写代码规则”固定下来，避免后面越写越乱。

### CubeMX 配置

- 新建或确认 STM32F103C8T6 工程可正常生成
- 配置系统时钟为后续常用方案
- 确认 `main.c`、`gpio.c`、`usart.c`、`adc.c`、`i2c.c` 等文件正常生成

### 代码存放位置

- 规则说明先保存在本文件
- 后续阶段记录保存到 `requirements/实施记录/`

### 编写逻辑

- 先明确哪些代码写在 `main.c`
- 哪些代码未来要抽到独立模块
- 先形成统一习惯，再开始功能开发

### 验收标准

- 明确后续每阶段都只做一个小目标
- 明确后续每阶段都保存一份记录

### 给 AI 的指令

```text
请不要直接实现全部功能。先按 STM32CubeMX 可重复生成代码的前提，给我制定一个最小、安全、适合初学者的代码组织规则。要求说明哪些代码放 main.c 的 USER CODE 区域，哪些代码适合后续抽成 app_xxx.c/app_xxx.h。
```

## 阶段 1：GPIO 点灯入门

### 目标

先完成最简单、最容易验证的功能：让一个 LED 周期闪烁。

推荐顺序：

1. 先只点亮 `PA8` 绿灯
2. 再扩展到 `PB0` 红灯、`PB1` 蓝灯

### CubeMX 配置

- 配置 `PA8` 为 `GPIO_Output`
- 可选：同时配置 `PB0`、`PB1` 为 `GPIO_Output`
- 输出速度使用低速或默认即可

### 代码存放位置

- `MX_GPIO_Init()` 由 CubeMX 自动生成
- 点灯逻辑写在 `main.c`
  - 初始化后的 `/* USER CODE BEGIN 2 */`
  - 主循环内的 `/* USER CODE BEGIN 3 */`

### 编写逻辑

- 初始化完成后进入死循环
- 循环中对 LED 引脚翻转
- 配合 `HAL_Delay()` 观察现象
- 这是最适合熟悉 HAL GPIO 接口的阶段

### 验收标准

- 绿灯能按固定周期闪烁
- 你能看懂 `HAL_GPIO_WritePin()` 与 `HAL_GPIO_TogglePin()` 的区别

### 给 AI 的指令

```text
我正在做 STM32F103C8T6 + CubeMX + HAL 的入门练习。请只帮我实现最小功能：PA8 绿灯闪烁。要求：
1. 告诉我 CubeMX 里要怎么配
2. 告诉我代码写到 main.c 的哪个 USER CODE 区域
3. 用初学者能看懂的方式解释每一行代码的作用
4. 不要提前引入状态机、任务调度或复杂封装
```

---

## 阶段 2：按键输入与轮询消抖

### 目标

学会读取按键输入，并通过按键控制 LED 或蜂鸣器。

建议先做：

- 按下 `PB12`，切换绿灯状态

### CubeMX 配置

- `PB12` 配置为 `GPIO_Input`
- 上拉输入 `Pull-up`
- 暂时不启用外部中断

### 代码存放位置

- 按键读取逻辑放在 `main.c` 的 `/* USER CODE BEGIN 3 */`
- 按键状态变量可放在 `/* USER CODE BEGIN PV */`

### 编写逻辑

- 主循环里轮询按键状态
- 判断按键是否按下
- 加入最简单的延时消抖
- 避免“一次按下触发多次切换”

### 验收标准

- 按下按键后 LED 只切换一次
- 松手后不会自动连跳

### 给 AI 的指令

```text
请在 STM32 HAL 工程里，帮我实现一个按键轮询案例：PB12 按下后切换 PA8 绿灯状态。要求：
1. 先说明 CubeMX 中按键输入该如何配置
2. 使用最基础的轮询 + 简单消抖
3. 告诉我变量该放在哪个 USER CODE 区域
4. 解释为什么会出现按一次触发多次的问题
```

---

## 阶段 3：外部中断与中断优先级入门

### 目标

把“轮询按键”升级成“按键中断”，开始真正理解中断写法。

建议先做：

- `PB13` 触发外部中断
- 中断发生时只置位一个事件标志
- 主循环里根据标志去切换 LED

### CubeMX 配置

- `PB13` 配置为 `GPIO_EXTI`
- 选择下降沿触发
- 在 NVIC 中启用对应 EXTI 中断
- 先保持默认优先级，后续再练优先级差异

### 代码存放位置

- 中断回调写在 `main.c` 的 `/* USER CODE BEGIN 4 */`
- 事件标志变量放在 `/* USER CODE BEGIN PV */`
- 主循环处理逻辑放在 `/* USER CODE BEGIN 3 */`

### 编写逻辑

- 中断里不要直接写复杂业务
- 中断里只设置 `key_event_flag = 1`
- 主循环里检测标志并处理 LED 切换

### 验收标准

- 按键响应比轮询更直接
- 你能理解“中断只报信，主循环干活”的基本思想

### 给 AI 的指令

```text
请帮我做一个 STM32 外部中断入门案例：PB13 按下触发中断，在主循环中切换 PA8 LED。要求：
1. 说明 CubeMX 里 EXTI 和 NVIC 怎么配置
2. 中断回调函数写在哪个 USER CODE 区域
3. 中断里不要做复杂操作，只做最短逻辑
4. 解释为什么中断里不建议长时间延时或打印串口
```

## 阶段 4：蜂鸣器与继电器输出控制

### 目标

学会控制真实执行器，包括蜂鸣器、水泵、风扇。

建议顺序：

1. 控制 `PA3` 蜂鸣器
2. 控制 `PA1` 水泵继电器
3. 控制 `PA2` 风扇继电器

### CubeMX 配置

- `PA3` 配置为 `GPIO_Output`
- `PA1` 配置为 `GPIO_Output`
- `PA2` 配置为 `GPIO_Output`

### 代码存放位置

- 先在 `main.c` 的 `/* USER CODE BEGIN 3 */` 验证
- 验证通过后，建议抽离：
  - `Core/Inc/app_actuator.h`
  - `Core/Src/app_actuator.c`

### 编写逻辑

- 先搞清楚继电器高电平有效还是低电平有效
- 用单独函数封装：
  - `Pump_On() / Pump_Off()`
  - `Fan_On() / Fan_Off()`
  - `Buzzer_On() / Buzzer_Off()`

### 验收标准

- 蜂鸣器可控
- 水泵继电器可控
- 风扇继电器可控
- 你知道执行器控制不能写反电平

### 给 AI 的指令

```text
请帮我基于 STM32 HAL 实现蜂鸣器、继电器输出控制的最小示例。使用 PA3 控制蜂鸣器，PA1 控制水泵继电器，PA2 控制风扇继电器。要求：
1. 先用最简单的 GPIO 输出方式
2. 帮我设计基础开关函数
3. 告诉我怎样判断继电器是高电平有效还是低电平有效
4. 先不要接入自动控制逻辑
```

## 阶段 5：UART1 调试输出与 ESP8266 通信准备

### 目标

先把串口调试打通，再考虑和 ESP8266 的正式交互。

建议先做：

- 串口发送固定字符串
- 串口接收单字节回显

### CubeMX 配置

- 开启 `USART1`
- `PA9` 为 TX，`PA10` 为 RX
- 波特率先设为 115200
- 8N1

### 代码存放位置

- 串口测试逻辑可先放在 `main.c`
- 稳定后建议抽离：
  - `Core/Inc/app_uart.h`
  - `Core/Src/app_uart.c`

### 编写逻辑

- 先会发，再会收
- 先阻塞式 `HAL_UART_Transmit()` / `HAL_UART_Receive()`
- 后续再升级为中断或 DMA

### 验收标准

- 串口助手能收到 STM32 发出的字符串
- 串口回显能工作

### 给 AI 的指令

```text
请帮我做 STM32F103C8T6 的 USART1 入门示例，PA9/PA10，波特率 115200。要求：
1. 先完成最简单的串口发送字符串
2. 再完成一个阻塞式单字节接收回显示例
3. 代码要适合初学者理解
4. 解释 HAL_UART_Transmit 和 HAL_UART_Receive 的基本用法
```

## 阶段 6：ADC 土壤湿度采样

### 目标

读取土壤湿度模拟量，为后面自动浇水提供基础数据。

### CubeMX 配置

- `PA0` 配置为 `ADC1_IN0`
- 开启 ADC1
- 先用单通道、软件触发
- 数据对齐右对齐

### 代码存放位置

- 初版采样逻辑可写在 `main.c`
- 稳定后建议抽离：
  - `Core/Inc/app_adc.h`
  - `Core/Src/app_adc.c`

### 编写逻辑

- 启动 ADC
- 等待转换完成
- 读取 ADC 原始值
- 先不要急着换算成“百分比”
- 先看原始值是否稳定、方向是否正确

### 验收标准

- 能稳定输出 ADC 原始值
- 干土和湿土读数有明显变化

### 给 AI 的指令

```text
请帮我实现 STM32 HAL 的 ADC 单通道采样示例，使用 PA0 读取土壤湿度传感器 AO。要求：
1. 先输出原始 ADC 值
2. 暂时不要做复杂滤波
3. 解释 ADC 初始化、启动、等待转换、读取结果的流程
4. 告诉我如何判断传感器输出方向是否与湿度变化一致
```

## 阶段 8：DS18B20 温度采集

### 代码存放位置

- 发起温度转换

### 验收标准

- 能读出稳定温度
- 变化趋势基本正确

### 给 AI 的指令

```text
请帮我基于 STM32 HAL + GPIO 软件时序，在 PA4 上实现 DS18B20 的最小驱动。要求：
```

## 阶段 9：阈值判断与联动控制逻辑

### 目标

开始把各个输入输出连起来，形成最小业务逻辑。

建议先做三条规则：

1. 土壤湿度过低时，启动水泵
2. 温度过高时，启动风扇
3. 异常时，蜂鸣器报警 + 红灯提示

### CubeMX 配置

- 本阶段主要使用前面已经配置好的外设
- 无需新增复杂外设

### 代码存放位置

- 建议新建：
  - `Core/Inc/app_control.h`
  - `Core/Src/app_control.c`

### 编写逻辑

- 不要把所有判断直接塞进 `while(1)`
- 先设计简单状态变量
- 用“采集 -> 判断 -> 执行器控制”的顺序写

### 验收标准

- 传感器变化能触发对应执行器动作
- 控制规则是清晰可读的，而不是一堆混乱 `if`

### 给 AI 的指令

```text
请基于我已经完成的 GPIO、ADC、DS18B20、蜂鸣器、继电器控制，帮我设计最小联动控制逻辑。要求：
1. 先不要上 RTOS
2. 用主循环 + 状态判断方式实现
3. 规则包括低湿度开水泵、高温开风扇、异常蜂鸣器报警
4. 帮我把代码结构设计得清晰，适合后续继续扩展
```

## 7. 推荐实际推进顺序

1. 阶段 1 GPIO 点灯
2. 阶段 2 按键轮询
3. 阶段 3 EXTI 中断
4. 阶段 4 蜂鸣器与继电器

## 8. 后续协作方式

后面我们可以严格按以下节奏一起推进：

1. 你指定当前要做的阶段，例如“开始阶段 1”
2. 我只为当前阶段提供：
   - CubeMX 具体配置步骤
   - 代码写入位置
   - 代码实现
   - 逐行解释
   - 验收方法
3. 当前阶段完成后，再把逻辑整理到 `requirements/实施记录/阶段xx-xxx.md`

这样做的好处是：

- 你不会一下被太多代码淹没
- 项目记录会越来越完整
- 后面即使重新接手，也知道每一步是怎么做出来的
