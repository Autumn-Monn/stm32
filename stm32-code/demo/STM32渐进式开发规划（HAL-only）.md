# STM32 渐进式开发方案（HAL-only）

## 1. 文档定位

本文档用于将本项目的 STM32 固件开发过程重构为一套可执行、可验证、可记录的渐进式开发方案。

目标不是一次性把系统做完，而是按照“单外设打通 -> 单功能稳定 -> 多模块联动 -> 系统集成”的顺序推进，确保每个阶段都具备以下特征：

- 能独立编译、下载、运行
- 有明确的验证方法
- 有可记录的阶段成果
- 为下一阶段保留清晰扩展接口

本文档是 `demo` 层面的执行方案，开发总控仍以以下文档为准：

- `requirements/03-渐进式开发任务清单.md`
- `requirements/02-STM32F103C8T6引脚分配.md`
- `requirements/实施记录/README.md`

## 2. 适用范围

本方案适用于以下条件：

- 主控：`STM32F103C8T6`
- 开发方式：`STM32CubeMX + HAL`
- 工程原则：`HAL-only`
- 编码环境：CubeMX 生成基础工程后，在 `USER CODE BEGIN/END` 或自建 `app_*.c/.h` 中编写业务逻辑
- 当前目标：先完成本地闭环控制与串口调试，不引入 RTOS，不引入复杂中间件，不追求一步到位联网架构

本方案明确排除以下内容：

- 直接使用标准库或裸寄存器重写底层驱动
- 在早期阶段引入 FreeRTOS
- 在未完成单模块稳定验证前做“大集成”
- 在没有验证记录的情况下跨阶段跳跃开发

## 4. 开发约束与总原则

### 4.1 HAL-only 原则

- 外设初始化统一由 CubeMX 生成
- 外设使用优先调用 HAL API
- 业务逻辑不直接改写 CubeMX 自动生成函数主体
- 如需自定义逻辑，优先放在：
  - `main.c` 的 `USER CODE BEGIN/END`
  - 自建 `app_xxx.c/.h`

### 4.2 渐进式推进原则

- 每阶段只解决一类主要问题
- 每次新增外设前，先确认前一阶段稳定
- 每一阶段必须形成“现象可见”的验证结果
- 未验证通过，不进入下一阶段

### 4.3 可维护性原则

- 早期允许在 `main.c` 快速验证
- 一旦某模块验证稳定，应逐步迁移到 `app_*.c/.h`
- 变量、状态、阈值要命名清楚，避免魔法数字散落
- 串口日志应成为系统调试的重要观察窗口

### 4.4 记录驱动原则

- 每完成一个阶段，必须在 `requirements/实施记录/` 下补充阶段记录
- 记录至少包括：
  - 阶段目标
  - 实际完成内容
  - 关键代码位置
  - 验证现象
  - 遇到的问题
  - 下一阶段准备事项

## 5. 工程组织建议

### 5.1 推荐目录职责

```text
Core/
  Inc/
  Src/
Drivers/
demo/
requirements/
```

建议随着阶段推进，逐步形成如下业务文件：

```text
Core/Inc/app_led.h
Core/Src/app_led.c
Core/Inc/app_key.h
Core/Src/app_key.c
Core/Inc/app_beep.h
Core/Src/app_beep.c
Core/Inc/app_relay.h
Core/Src/app_relay.c
Core/Inc/app_uart.h
Core/Src/app_uart.c
Core/Inc/app_soil.h
Core/Src/app_soil.c
Core/Inc/app_temp.h
Core/Src/app_temp.c
Core/Inc/app_control.h
Core/Src/app_control.c
```

### 5.2 代码放置边界

CubeMX 自动生成内容主要负责：

- 时钟
- GPIO 初始化
- UART 初始化
- ADC 初始化
- I2C 初始化
- 中断向量与 HAL 框架

用户代码主要负责：

- 模块状态机
- 外设业务封装
- 按键事件处理
- 采样与滤波
- 阈值判断
- 联动控制策略
- 调试输出

### 5.3 迁移建议

- 阶段 1~3：允许大部分验证代码先放在 `main.c`
- 阶段 4~6：开始把稳定模块拆到 `app_*.c`
- 阶段 7~10：尽量做到 `main.c` 只保留调度入口，业务逻辑转移到模块文件

## 阶段0：最小开发规范

### 目标

- 建立统一的 CubeMX/HAL 开发边界
- 明确引脚使用、命名和记录要求
- 建立一个可持续迭代的工程基线

### 前置条件

- 已确认芯片型号、开发板供电、下载器可用
- 已准备项目目录结构
- 已确认 `requirements/02-STM32F103C8T6引脚分配.md`

### CubeMX 配置要求

- 选择正确 MCU：`STM32F103C8Tx`
- 配置系统时钟为项目约定值
- 保持下载接口与调试接口可用
- 先不打开无关外设

### 实施步骤

1. 新建 CubeMX 工程，确认时钟树与基础下载设置正确。
2. 生成 HAL 工程，确保工程首次可编译。
3. 在 `main.c` 中保留最少量用户入口逻辑。
4. 建立阶段记录机制，准备 `requirements/实施记录/阶段00-最小开发规范.md`。

### 验证方法

- 工程可成功生成
- 工程可成功编译
- 固件可下载到板子
- 芯片能正常运行，不出现下载失败、复位异常、死机

### 常见风险

- 时钟配置错误导致串口、延时异常
- 调试接口被误占用
- 自动生成代码与手写代码边界混乱

### 进入下一阶段条件

- 已有一个可重复生成、可编译、可下载的基线工程

## 阶段1：GPIO 点灯

### 目标

- 打通最基本的 GPIO 输出
- 确认 LED 引脚、电平逻辑和 HAL 延时可用

### 前置条件

- 阶段0 已完成

### CubeMX 配置要求

- 配置 `PA8`、`PB0`、`PB1` 为 GPIO Output
- 输出模式采用推挽输出
- 默认输出电平按硬件实际情况设定

### 代码落点建议

- 初版可先放在 `main.c`
- 稳定后迁移到 `app_led.c/.h`

### 实施步骤

1. 编写单灯闪烁逻辑。
2. 再扩展为三灯顺序闪烁。
3. 整理 LED 的开、关、翻转接口。

### 验证方法

- 绿灯、红灯、蓝灯均可独立点亮与熄灭
- 能观察到固定周期闪烁
- 三个 LED 不出现互相干扰

### 常见风险

- LED 高低电平有效逻辑与硬件不一致
- `HAL_Delay()` 表现异常，通常反映时钟配置问题

### 进入下一阶段条件

- 已确认 GPIO 输出链路稳定

---

## 阶段2：按键轮询

### 目标

- 打通 GPIO 输入读取
- 通过轮询实现按键检测与基本消抖

### 前置条件

- 阶段1 已完成

### CubeMX 配置要求

- 配置 `PB12~PB15` 为 GPIO Input
- 根据硬件接法选择上拉或下拉

### 代码落点建议

- 初版可在 `main.c` 验证
- 稳定后拆分为 `app_key.c/.h`

### 实施步骤

1. 编写单个按键读取逻辑。
2. 增加软件消抖。
3. 扩展为 4 个按键统一扫描。
4. 建立“按下事件”与“持续按下状态”的区分。

### 验证方法

- 每个按键都能稳定识别
- 单次按下只触发一次动作
- 长按不会抖动触发多个误动作

### 推荐验证案例

- K1 控制绿灯翻转
- K2 控制红灯翻转
- K3 控制蓝灯翻转
- K4 控制全部熄灭

### 常见风险

- 未做消抖导致一次按下触发多次
- 输入上下拉方向与实际电路不一致

### 进入下一阶段条件

- 按键轮询机制稳定，可作为后续的人机输入基础

---

## 阶段3：EXTI 中断

### 目标

- 掌握按键中断触发方式
- 明确轮询与中断的边界

### 前置条件

- 阶段2 已完成

### CubeMX 配置要求

- 选择 1~2 个按键配置为 EXTI 模式
- 配置触发边沿
- 打开对应 EXTI 中断

### 代码落点建议

- 中断回调放在 HAL 规定位置
- 业务动作不要直接写在中断内，建议仅置标志位

### 实施步骤

1. 配置一个按键为 EXTI。
2. 在 `HAL_GPIO_EXTI_Callback()` 中记录事件。
3. 主循环处理事件并驱动 LED 或蜂鸣器响应。

### 验证方法

- 按下指定按键后可立即响应
- 中断中断点可命中
- 响应逻辑不会阻塞系统

### 常见风险

- 在中断里做过多处理，导致系统卡顿
- 忽略抖动，导致中断重复进入

### 进入下一阶段条件

- 已具备基础中断事件处理能力

---

## 阶段4：蜂鸣器与继电器

### 目标

- 打通执行器类数字输出
- 明确蜂鸣器、继电器的控制抽象接口

### 前置条件

- 阶段1 稳定
- 建议阶段2 已具备按键输入控制能力

### CubeMX 配置要求

- `PA1` 配置为水泵继电器输出
- `PA2` 配置为风扇继电器输出
- `PA3` 配置为蜂鸣器输出

### 代码落点建议

- `app_beep.c/.h`
- `app_relay.c/.h`

### 实施步骤

1. 编写蜂鸣器开关函数。
2. 编写继电器开关函数。
3. 验证单独控制。
4. 增加简单提示动作，例如“按键触发时蜂鸣器短鸣一次”。

### 验证方法

- 蜂鸣器可按预期鸣叫/静音
- 水泵继电器和风扇继电器可独立吸合/断开
- LED、按键、蜂鸣器、继电器可协同运行

### 常见风险

- 继电器有效电平判断错误
- 继电器吸合时电源波动影响主控
- 蜂鸣器和继电器与 LED 共享逻辑时出现状态混乱

### 进入下一阶段条件

- 已形成统一的执行器控制接口

## 阶段5：UART 调试

### 目标

- 建立系统的基础调试输出通道
- 将“肉眼看灯”升级为“串口看状态”

### 前置条件

- 阶段1~4 至少完成基础验证

### CubeMX 配置要求

- 打开 `USART1`
- `PA9` 为 TX，`PA10` 为 RX
- 波特率建议统一为 `115200`

### 代码落点建议

- `app_uart.c/.h`
- 可以额外提供 `debug_printf()` 或 `uart_log()` 接口

### 实施步骤

1. 打通串口发送字符串。
2. 输出系统启动日志。
3. 输出按键、LED、继电器状态变化日志。
4. 如有需要，补充最小接收能力。

### 验证方法

- 上电可看到启动信息
- 按键触发时可看到串口日志
- 继电器动作、ADC 数值、温度数据都能通过串口打印

### 常见风险

- 串口乱码，通常与波特率或时钟设置相关
- 在阻塞发送中打印过多导致系统卡顿

### 进入下一阶段条件

- 系统关键状态可通过串口输出观察

---

## 阶段6：ADC 土壤湿度

### 目标

- 打通模拟量采集链路
- 能稳定读取土壤湿度传感器原始值并形成基础判断

### 前置条件

- 阶段5 已完成，串口可用

### CubeMX 配置要求

- `PA0` 配置为 ADC 输入
- 打开 ADC 外设
- 优先使用单通道、单次转换方式进行入门验证

### 代码落点建议

- `app_soil.c/.h`

### 实施步骤

1. 完成原始 ADC 采样。
2. 将采样值打印到串口。
3. 加入多次采样平均。
4. 引入干湿阈值与分级判断。

### 验证方法

- 干燥、潮湿状态下数值明显变化
- 串口能稳定输出采样结果
- 多次读取波动在可接受范围内

### 推荐输出内容

- 原始 ADC 值
- 平均值
- 干湿判定结果

### 常见风险

- 传感器电源与模拟地噪声大
- 直接使用单次原始值导致抖动明显
- 阈值未经标定，判断不可靠

### 进入下一阶段条件

- 已得到可用于联动控制的基础湿度数据

### 目标

- 建立 I2C 通讯能力
- 为后续可扩展传感器模块预留接口

### 前置条件

- 阶段5 稳定

### CubeMX 配置要求

- 打开 `I2C1`

### 代码落点建议

- `app_i2c.c/.h` 或在具体模块中封装

### 实施步骤

2. 用一段最小读写流程验证总线可工作。

### 验证方法

- 能完成至少一次设备探测或寄存器访问
- 出错时能通过串口打印错误信息

### 常见风险

- 上拉缺失或不合理
- 地址位理解错误
- 时序配置过快导致不稳定

### 进入下一阶段条件

- I2C 通讯链路已具备可复用能力

## 阶段8：DS18B20 温度采集

### 目标

- 打通温度传感器采集链路
- 输出可用于控制逻辑的温度数据

### 前置条件

- 阶段5 已完成
- 已确认 `PA4` 分配给 DS18B20

### CubeMX 配置要求

- `PA4` 作为 GPIO，供单总线协议实现使用

### 代码落点建议

- `app_temp.c/.h`
- 若实现较长，可拆为 `bsp_ds18b20.c/.h`

### 实施步骤

1. 实现复位、存在脉冲检测。
2. 实现读写字节流程。
3. 完成温度转换与读取。
4. 将温度值通过串口输出。
5. 做基本容错，例如无器件时给出错误提示。

### 验证方法

- 上电后可稳定读取温度
- 加热/降温时温度读数跟随变化
- 读数格式清晰，便于控制逻辑调用

### 常见风险

- 单总线时序不稳定
- 读到固定值或异常值
- 未处理器件缺失导致程序阻塞

### 进入下一阶段条件

- 温度数据已具备稳定读取能力

## 阶段9：联动控制

### 目标

- 将采集模块与执行模块联动起来
- 建立本地自动控制闭环

### 前置条件

- 湿度采集稳定
- 温度采集稳定
- 继电器控制稳定
- 串口日志可观察

### 代码落点建议

- `app_control.c/.h`

### 推荐控制策略

- 湿度低于下限：启动水泵
- 湿度恢复到上限：关闭水泵
- 温度高于上限：启动风扇
- 温度回落到下限：关闭风扇
- 异常状态：蜂鸣器提示 + 串口报警

建议加入滞回区，避免边界抖动导致执行器频繁切换。

### 实施步骤

1. 定义控制阈值与状态结构体。
2. 实现采集值到控制决策的映射。
3. 增加模式区分：
   - 手动模式
   - 自动模式
4. 输出联动日志。

### 验证方法

- 人为模拟干燥状态时水泵能启动
- 人为模拟高温状态时风扇能启动
- 条件恢复后执行器能正确关闭
- 边界附近不会频繁抖动开关

### 常见风险

- 采样噪声导致误动作
- 无滞回导致继电器频繁吸合
- 手动与自动模式优先级不清晰

### 进入下一阶段条件

- 已形成最小可用本地闭环控制系统

## 8. 统一开发节奏建议

每个阶段建议都遵循以下节奏：

1. 先确认目标，不同时做多个新模块。
2. 再完成 CubeMX 最小配置。
3. 然后用最少代码打通现象。
4. 打通后补充串口日志。
5. 稳定后再抽象为模块接口。
6. 最后补实施记录并进入下一阶段。

## 9. 阶段验收标准（Definition of Done）

每一阶段完成时，至少满足以下条件：

- 工程可编译
- 固件可下载
- 功能现象可重复观察
- 有串口或外在现象可验证
- 已记录关键代码位置
- 已记录当前阶段的问题与遗留事项

如果上述任一项未满足，则该阶段不应视为完成。

## 11. AI 协作约束

如果后续借助 AI 辅助编写 STM32 代码，必须明确以下约束：

- 必须基于 `HAL-only`
- 不允许擅自改动 CubeMX 自动生成区之外的初始化结构
- 新增代码优先写成模块文件，不要持续堆积在 `main.c`
- 每次只处理一个阶段的问题
- 生成代码时要包含“放在哪个文件、哪个函数、如何验证”

推荐提问模板：

```text
当前阶段：阶段6 ADC 土壤湿度
约束：STM32F103C8T6，HAL-only，CubeMX 已生成 ADC1
目标：读取 PA0 的土壤湿度值，并通过 USART1 打印
要求：
1. 说明代码放在哪个文件
2. 说明需要调用哪些 HAL API
3. 提供验证步骤
4. 不要引入寄存器写法
```

## 12. 本文档的使用方式

开发时建议把本文档当作“执行清单”而不是“阅读材料”：

- 开始阶段前，先看该阶段的目标与前置条件
- 配置工程时，只做该阶段要求的最小配置
- 编码时优先完成最小可见现象
- 验证通过后再整理抽象
- 最后补实施记录

这样做的结果是：即使系统还没有全部完成，我们也始终拥有一个稳定、可运行、可回退、可继续扩展的 HAL-only 固件基线。
